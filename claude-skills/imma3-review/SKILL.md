# imma3-review

**Announce:** "Using imma3-review to review, fix, and capture learnings."

**Thinking mode:** Use ultrathink (extended thinking) for the review. Catching subtle agent mistakes requires careful cross-referencing.

## Key Terms

- **Scope**: Top-level grouping directory under `claude_detailed_plans/`. Derived from the task file's `Scope` field or directory structure. Legacy `Submodule:` or `Module:` fields are equivalent.
- **complete/**: Archive directory at `claude_detailed_plans/complete/`. Finished features are moved here with date stamps.
- **Learnings**: Accumulated lessons at `claude_detailed_plans/learnings/`. The primary record of edge cases, gotchas, and corrections.

## Prerequisites

**Required:** A `CLAUDE.md` at the project root (or submodule root) describing tech stack, conventions, and build/check command(s).

**If `CLAUDE.md` is missing:** Stop and tell the user: "This project needs a `CLAUDE.md` before imma3-review can run."

---

## Phase 1: Orient

### Detect Stack and Conventions

1. **Read the project's CLAUDE.md** for conventions, architecture, and build commands.
2. **Detect the stack** from config files (same detection as imma3-plan/execute).
3. **Read the execution report** from `claude_detailed_plans/<scope>/results/` — this tells you what was built and what files were touched.
4. **Read the EXECUTION_TRACKER.md** for overall status.
5. **Read existing learnings** in `claude_detailed_plans/learnings/` if any — avoid flagging things already addressed.

### Identify Scope

From the execution report and/or user input, build the list of:
- Files created (need full review)
- Files modified (need diff-aware review)
- Build check results (existing issues to address)

---

## Phase 2: Gather Issues

### If the user has tested and reports issues:

Collect everything the user describes before starting fixes. They may provide:
- A verbal list of issues
- Screenshots or error messages
- A file with notes
- A mix of bugs, convention violations, missing behavior, and edge cases

### If this is a first-pass review (no user testing yet):

Proceed directly to code review.

---

## Phase 3: Code Review

### File-by-File Review

Read EVERY file from the execution report. For each:

1. **Does it exist?** If claimed as created and missing, that's a critical failure.

2. **Does it follow project conventions?** Check against CLAUDE.md and the template feature patterns. Common issues:
   - Wrong naming conventions
   - Missing or incorrect imports
   - Wrong state management pattern
   - Hard-coded values that should use project constants/tokens
   - Missing error handling at system boundaries
   - Incorrect API/data paths

3. **Is it complete?** Check that all fields, methods, and handlers specified in the task file are present. Agents frequently:
   - Implement 80% of a model and miss fields in serialization methods
   - Create a component but forget to register/export it
   - Add a route handler but miss the route registration
   - Skip edge cases mentioned in the task

4. **Cross-file consistency:**
   - Types/models used consistently across all files that reference them
   - Imports resolve to real exports
   - Routes are reachable from navigation
   - New items added to barrel files / indexes where needed

### Duplication Scan

For each new file:
1. Catalog all functions, types, components, or classes it defines
2. Search the codebase for similar names or identical logic
3. Categorize:

| Finding | Action |
|---------|--------|
| Private item used only in its own file | Leave in place |
| Same logic duplicated across 2+ files | Extract to shared location |
| Generic utility that belongs in common/shared | Move it |
| One-time pattern | Leave — don't create premature abstractions |

---

## Phase 4: Triage

If the user reported issues, categorize each:

| Category | Action |
|----------|--------|
| **Bug** — code doesn't do what the plan said | Fix directly |
| **Convention violation** — code works but doesn't follow project patterns | Fix directly |
| **Missing behavior** — plan didn't account for this case | Fix if small; if large, note it needs a new task |
| **Edge case** — works for happy path but fails on edge | Fix directly |
| **Design issue** — the plan itself was wrong | Discuss with user before fixing |

Present the triage: "Here's what I found. N bugs, N convention issues, N edge cases, N from code review. Shall I fix all of them?"

**Do NOT proceed without confirmation.**

---

## Phase 5: Fix

For small/medium fixes: fix directly, reading each file before modifying.

For large fixes spanning multiple files: dispatch fix agents using the **Executor** template from `/imma3-execute`'s `agents.md`:
- Fill `{{KNOWN_GOTCHAS}}` with the user's issue description for that area
- Fill `{{TASK_CONTENTS}}` with a concise fix description (what's wrong, what it should be, which files)

After all fixes:
1. Run the build/check command from CLAUDE.md
2. List all files modified with a one-line summary per file

---

## Phase 6: Generate Review Report

Write to `claude_detailed_plans/<scope>/results/YYYY-MM-DD_review_report.md`:

```markdown
# Code Review Report

**Date:** YYYY-MM-DD
**Scope:** <scope>
**Feature:** <feature>
**Stack:** <detected stack>
**Files Reviewed:** N
**Issues Found:** N
**Issues Fixed:** N

## Summary

### Critical Issues (blocking)
1. [file:line] Description — **Fixed** / **Needs follow-up**

### Convention Violations
1. [file:line] Description — **Fixed**

### Duplication Found
1. Description — **Extracted to** <new location> / **Left in place** (reason)

### Missing Implementations
1. Description — **Implemented** / **Created follow-up task**

### Clean Files (no issues)
- file — all checks passed

## Build Check
**Before fixes:** [pass/fail with details]
**After fixes:** [pass/fail]

## Follow-up Tasks
[Any new TASK files created for unresolved issues]
```

## Phase 7: Create/Update Learnings

This is the most important long-term output. Learnings accumulate over time and make future imma3-plan and imma3-execute runs better.

**If no lessons were discovered (all issues were trivial typos), skip — do not create empty files.**

Write to `claude_detailed_plans/learnings/<scope>_<feature>_<date>.md`:

```markdown
# Learnings: [Feature Name]

**Date:** YYYY-MM-DD
**Scope:** <scope>
**Stack:** <detected stack>

## What Went Well
- [Things agents got right — patterns that worked, good task breakdown decisions]

## What Went Wrong
- [Issues found — be specific about the mistake and the fix]

## Patterns Discovered
- [New patterns observed that future plans should follow]
- [Include file paths as references]

## Agent Pitfalls
- [Specific mistakes agents made that should be anticipated in future task files]
- [e.g., "Agents consistently forget to add new providers to the barrel export file"]

## Convention Reminders
- [Conventions from CLAUDE.md that were violated — restate for emphasis]
- [Any conventions NOT in CLAUDE.md that should be added]

## Recommendations
- [Suggestions for CLAUDE.md updates]
- [Suggestions for future task file improvements]
```

### Learnings Index

After writing, update (or create) `claude_detailed_plans/learnings/INDEX.md`:

```markdown
# Learnings Index

| Date | Scope | Feature | Key Takeaway | File |
|------|-------|---------|--------------|------|
```

### Learnings Rules

- Group entries by scope, then by category
- Keep entries concise but include a concrete remedy — 1-5 sentences plus optional bullet
- Remove entries older than 90 days or no longer relevant (fixed upstream, convention changed)
- If a lesson recurs 3+ times, suggest the user promote it to `CLAUDE.md` as a permanent convention
- Deduplicate — if a new lesson describes the same root cause as an existing entry, merge them (update date, add feature name, refine the remedy)
- Use scope `general` for lessons that apply across all scopes

---

## Phase 8: Check with User

Ask: **"Fixes applied. Want to test again, or is this feature complete?"**

- **Test again** → user tests, comes back with more issues, repeat from Phase 2
- **Complete** → proceed to Archive

---

## Phase 9: Archive

Only when the user confirms the feature is complete.

Move the completed feature's plan directory:

```
claude_detailed_plans/<scope>/<feature>/
  → claude_detailed_plans/complete/<scope>/<feature> [DD-MM-YY]/
```

Date format for directory name: `DD-MM-YY` (completion date).

Steps:
1. `mkdir -p claude_detailed_plans/complete/<scope>`
2. Move the feature directory with date suffix
3. Move `results/` into the complete directory if present
4. Remove empty parent directories (`rmdir` only if empty — don't remove directories with other pending plans)
5. Update `MASTER_GUIDE.md` status to Complete
6. Report: what was archived and what remains pending

```
Archived:
  complete/<scope>/<feature> [DD-MM-YY]  — N tasks

Still pending:
  <scope>/<other-feature>                — N tasks
```

---

## Phase 10: Update State

After the review:
- **Update MASTER_GUIDE.md** — mark the feature as "Reviewed" or "Complete" in the Current Sprint table, add any decision log entries
- **Update task files** — mark reviewed tasks with updated status
- **Update EXECUTION_TRACKER.md** — note the review pass
- **Suggest CLAUDE.md updates** — if the review revealed conventions that should be documented in CLAUDE.md, tell the user and offer to add them. Present the specific lines to add. This closes the feedback loop: learnings → conventions → better future runs.

---

## Key Principles

- **NEVER commit to git** — the user manages their own git workflow
- **Read every file** — do not sample or skip files from the execution report
- **Fix, don't just report** — if the fix is clear, apply it directly
- **Learnings are the legacy** — the review report is for now, the learnings file is for all future work
- **Convention violations are patterns** — if agents make the same mistake twice, it's a task-writing issue, not just a code issue
- **Be surgical** — only change what genuinely needs fixing, don't refactor working code
- **Build must pass** — the review isn't done until the build/check command succeeds
- **User confirms before fixing** — always present the triage first
- **Archive on completion** — keep `claude_detailed_plans/` showing only active work
- **Learning is the goal** — every fix cycle should capture lessons that prevent recurrence

## Follow-up Commands

After review:
- `/imma3-plan` — plan the next feature (learnings will inform better plans)
- `/imma3-execute` — execute remaining tasks if any
